# TradePulse System Modules Overview

Формалізований огляд основних модулів TradePulse і пов'язаних артефактів. Документ допомагає дослідникам, інженерам і SRE швидко зрозуміти призначення системи, складові частини та те, як ними користуватися.

## Призначення системи
- **Мета:** забезпечити безпечну та масштабовану платформу для досліджень, бектестів і живої торгівлі з геометричними індикаторами.
- **Роль у життєвому циклі:** дослідження → моделювання → симуляція → розгортання → спостереження та контроль ризиків.
- **Ключові артефакти:** вимоги (docs/requirements), контракти інтерфейсів (docs/contracts/interface-contracts.md), ADR (docs/adr), експлуатаційні плейбуки (docs/runbook_*). Це модульний «паспорт» до цих артефактів.

## Аудиторія
- **Quant/Research:** швидко зрозуміти які дані та параметри потрібні для індикаторів і стратегій.
- **Platform/Backend:** знати, які сервіси інтегрувати, які API відкриваються, які SLO та інваріанти контролювати.
- **SRE/Operations:** мати прямі посилання на плейбуки та сигнали моніторингу для кожного модуля.

## Модулі та їх формалізація
Нижче наведено ключові модулі з чітким призначенням, інтерфейсами, залежностями та контрольними артефактами.

### 1. Adaptive Risk Manager
- **Призначення:** динамічно обмежує експозицію (VaR/CVaR, позиційні ліміти) і активує захисти (kill switch, circuit breakers) перед відправкою ордерів.
- **Інтерфейси:**
  - Вхід: `risk_check(request)` приймає профіль стратегії, позиційні дані й метрики волатильності (див. docs/contracts/interface-contracts.md, секція Pre-Trade Risk).
  - Вихід: рішення `ALLOW | BLOCK | REQUIRE_REVIEW` з причиною та рекомендаціями.
- **Залежності:** TACL політики (TACL.md), контроль якості даних (docs/risk_controls.md), журналювання безпеки.
- **Операції та артефакти:**
  - Плейбук інцидентів: docs/runbook_kill_switch_failover.md.
  - Спостереження: SLO латентності перевірки < 10 мс; алерти на частоту блокувань і SLA помилок інтеграцій.

### 2. Market Regime Analyzer
- **Призначення:** визначає режим ринку (тренд/флет/волатильність) за допомогою Hurst exponent, ADF-тестів та інших індикаторів, щоб адаптувати параметри стратегії.
- **Інтерфейси:**
  - Вхід: нормалізовані таймсерії з даними якості (docs/requirements/requirements-specification.md — REQ-002).
  - Вихід: структура `RegimeClassification` з типом режиму, впевненістю та рекомендованими параметрами (наприклад, довжина вікна, пороги волатильності).
- **Залежності:** бібліотека індикаторів (docs/indicators.md), синхронізація часових рядів (плановий ADR-0004), кешування результатів для бектестів.
- **Операції та артефакти:**
  - Плейбук деградацій даних: docs/runbook_data_incident.md.
  - Моніторинг: дрейф режимів, контроль «stale» класифікацій, SLO оновлення < 1 хвилини на новий бар.

### 3. Dynamic Position Sizer
- **Призначення:** розраховує розмір позиції на основі Kelly Criterion, волатильності та risk parity з урахуванням обмежень портфеля.
- **Інтерфейси:**
  - Вхід: `PositionContext` (ризиковий бюджет, історична волатильність, кореляції, поточні ліміти).
  - Вихід: рекомендовані обсяги/лот-розміри та граничні значення для stop-loss/take-profit.
- **Залежності:** Risk Manager для фінального схвалення, Market Regime Analyzer для режимно-залежних множників, модуль даних для ковзаючих вікон волатильності.
- **Операції та артефакти:**
  - Плейбук аномалій розміру позицій: docs/runbook_live_trading.md (секція про захисти виставлення ордерів).
  - Метрики: дельта між рекомендованим та фактичним розміром, відсоток відхилених рекомендацій, латентність < 5 мс.

### 4. Agent Coordinator
- **Призначення:** централізовано розподіляє ресурси між агентами (дослідницькі, торгові, дані), управляє задачами, конфліктами й SLA.
- **Інтерфейси:**
  - Вхід: події від планувальника або ручні запити (HTTP/gRPC), параметри пріоритетів і квот.
  - Вихід: оркестраційні рішення (`ASSIGN`, `THROTTLE`, `REQUEUE`), оновлення статусу агентів, сигнали зняття навантаження.
- **Залежності:** черги подій (docs/queue_and_backpressure.md), моніторинг агентів (docs/monitoring.md), політики безпеки доступу.
- **Операції та артефакти:**
  - Плейбук backpressure: docs/runbook_disaster_recovery.md (секція про деградацію сервісів даних) та docs/runbook_time_synchronization.md.
  - Метрики: час реакції на подію, коефіцієнт конфліктів за ресурси, відповідність SLA (< 100 мс на рішення).

## Як використовувати модулі
1. **Обрати точку інтеграції:** звернутися до contracts (docs/contracts/interface-contracts.md) для точних сигнатур API.
2. **Налаштувати дані:** забезпечити відповідність вимогам REQ-002 (якість та версіонування даних) перед запуском індикаторів або режимних моделей.
3. **Увімкнути контроль ризиків:** підключити Adaptive Risk Manager як обов'язковий шлюз для ордерів і стратегії.
4. **Перевірити операційні плейбуки:** переконатися, що команда SRE має швидкий доступ до релевантних runbook для кожного модуля.
5. **Верифікувати продуктивність:** звірити SLO/latency кожного модуля з метриками в `docs/performance.md` і `docs/reliability.md` перед випуском.

## Карта артефактів
- **Вимоги → Реалізація:** REQ-001/002/003/005 покриваються модулями 1–4 відповідно (див. docs/requirements/requirements-specification.md).
- **Рішення → Архітектура:** ADR-0001, ADR-0002 та планові ADR-0004/0005/0006 визначають інваріанти для відповідних модулів.
- **Операції → Runbooks:** для кожного модуля зазначено основний плейбук; додаткові сценарії у docs/incident_playbooks.md та docs/operations.md.

## Незалежні пакети моделей

Для спрощення еволюції монорепозиторію частина моделей тепер формалізована як
окремі installable-пакети з локальними `pyproject.toml`:

- `analytics/fpma` → `tradepulse-fpma`
- `analytics/regime` → `tradepulse-regime`
- `markets/orderbook` → `tradepulse-orderbook`
- `nak_controller` → `tradepulse-nak`
- `neurotrade_pro` → `neurotrade-pro`

Детальний план розділення та межі між моделями наведені в
`docs/architecture/independent_models.md`.

Цей огляд підтримує трасування від бізнес-вимог до реалізації та експлуатаційних артефактів, забезпечуючи спільне розуміння системи для всіх ролей.
