# Стратегія RLHF/RLAIF для TradePulse

## 1. Огляд програми
- **Мета**: розвинути послідовний процес навчання моделей TradePulse з використанням RLHF (Reinforcement Learning from Human Feedback) та RLAIF (Reinforcement Learning from AI Feedback).
- **Принципи**:
  1. Безпека та відповідність регуляторним вимогам фінансової галузі.
  2. Відтворюваність результатів і прозорість процесів прийняття рішень.
  3. Модульність та можливість швидкого масштабування робочих циклів.

## 2. Організація анотаційного процесу
### 2.1 Інструкції для анотації
- Створити основний посібник «RLHF Annotation Guide» з версіонуванням у репозиторії (`docs/annotation/guide_v{major}.{minor}.md`).
- У посібнику визначити:
  - **Цілі** (наприклад, оцінка торгових рекомендацій, пояснень, сценаріїв ризику).
  - **Вхідні дані** (ринкові дані, звіти ризиків, коментарі трейдерів).
  - **Категорії оцінки**: точність, відповідність політикам, прозорість мотивації, читабельність.
  - **Правила маркування**: приклади «правильних» та «неправильних» відповідей, типові помилки.
  - **Пріоритет безпеки**: обов’язкові тригери ескалації (наприклад, рекомендації, що порушують комплаєнс).
- Інструкції повинні бути двомовними (українська + англійська) та регулярно переглядатися командою безпеки.

### 2.2 Шаблони анотацій
- **Формат JSON/Markdown**:
  ```json
  {
    "prompt_id": "<UUID>",
    "prompt_text": "...",
    "model_response": "...",
    "annotator_id": "<ID>",
    "scores": {
      "accuracy": <0-5>,
      "policy_compliance": <0-5>,
      "risk_awareness": <0-5>,
      "communication": <0-5>
    },
    "labels": ["high_risk", "needs_revision"],
    "rationale": "...",
    "improvement_suggestions": "...",
    "timestamp": "ISO-8601"
  }
  ```
- Запровадити шаблони у вигляді Jinja2 для інтерфейсу анотації та YAML-схеми для валідації (`schemas/annotation_v1.yaml`).

### 2.3 Контроль якості анотацій
- **Подвійна перевірка**: 20% зразків отримують повторну анотацію незалежним експертом.
- **Вимірювання узгодженості**: щотижневий розрахунок Cohen’s Kappa та Krippendorff’s Alpha; поріг прийнятності ≥0.75.
- **Рев’ю аналітиків ризику**: автоматичні тригери для записів з низьким комплаєнсом.
- **Періодичні аудити**: щомісяця – вибірка 5% анотацій для глибинного розбору та оновлення інструкцій.

## 3. Багаторівневі оцінювачі
- **Рівень 0 – Автоматичний фільтр**: регулярні вирази, NLP-класифікатори та перевірки формату.
- **Рівень 1 – Молодший анотаційний шар**: оцінка базової якості та комплаєнсу.
- **Рівень 2 – Старший експерт**: фокус на ризики, економічну логіку, довготермінові наслідки.
- **Рівень 3 – Комітет**: колективне рішення для спірних випадків і оновлення політик.
- **RLAIF-рівень**: спеціалізовані мовні моделі, що дають попередні оцінки/рекомендації; вони проходять валідацію проти людських еталонів.

## 4. Активне навчання та добір даних
- **Механізм відбору**: використання невизначеності політики (entropy sampling), diversity sampling (k-means на ембедінгах) та бізнес-приорів (події високої волатильності).
- **Цикл**:
  1. Модель генерує відповіді на останні ринкові сценарії.
  2. Модуль відбору оцінює невизначеність/цінність даних.
  3. Вибрані приклади направляються анотаторам або RLAIF-агенту.
  4. Нові анотації інтегруються у буфер повторного навчання.
- **Моніторинг**: відстеження coverage метрик та ефективності (зменшення regret, підвищення Sharpe Ratio на симуляторах).

## 5. Само-критика та само-узгодження моделей
- Впровадити стадію «self-reflection» після кожної відповіді моделі: спеціальна модель або ланцюжок запитів генерує критику та план виправлення.
- Результати само-критики зберігаються у графі знань для подальшого аналізу повторюваних помилок.
- Регулярно порівнювати якість відповідей «до» і «після» само-рефлексії; вимірювати підвищення оцінок QA та зниження штрафів.

## 6. Штрафні функції та дизайн винагороди
- **Базова функція винагороди**:
  \[
  R = w_1 \cdot S_{accuracy} + w_2 \cdot S_{risk} + w_3 \cdot S_{communication} - P_{violations} - P_{uncertainty}
  \]
- **Компоненти штрафів**:
  - `P_violations`: великі штрафи за порушення політик (комплаєнс, токсичність, рекомендації без дисклеймерів).
  - `P_uncertainty`: штрафи за надмірну впевненість без підтримки даними.
  - Додаткові Lagrangian-покарання за порушення обмежень (наприклад, VaR, регуляторні ліміти).
- **Адаптивні ваги**: щомісячне переглядання коефіцієнтів на основі бізнес KPI.

## 7. Безпечні обмеження
- Формалізувати правила у вигляді машинно-зчитуваних політик (`configs/policies/*.yaml`).
- Використовувати constrained RL (наприклад, CPO, Lagrangian methods) для забезпечення:
  - Обмежень на Value-at-Risk, Expected Shortfall.
  - Заборонених рекомендацій (маніпуляції ринком, інсайдерські дані).
  - Безпечного тону комунікації (Reg BI, MiFID II).
- Перед тренуванням виконувати статичний аналіз промптів/відповідей на предмет заборонених патернів.

## 8. Симулятори користувачів
- **Ролі**: роздрібний клієнт, професійний трейдер, ризик-менеджер, служба комплаєнсу.
- **Модулі**:
  - Поведінкові моделі з історичних логів (RL-агенти, MARL).
  - Скрипти сценаріїв для рідкісних подій (кризові новини, black swan).
  - Параметризовані симулятори ринку (підсистема `backtest/` та `markets/`).
- **Використання**: генерування синтетичних діалогів та завдань для RLHF/RLAIF, stress-testing відповідів.

## 9. Scorecards та метрики
- **Human Feedback Scorecard**: дашборд (Superset/Metabase) з метриками середніх оцінок, дисперсій, Kappa.
- **Model Alignment Scorecard**: частота комплаєнс-порушень, precision/recall для ризикових тегів, Sharpe/Sortino у симуляторі.
- **Safety Scorecard**: кількість ескалацій, false negative/positive для критичних порушень, час реагування.
- **Автоматизовані алерти**: інтеграція з PagerDuty/Slack.

## 10. Навчальні цикли
### 10.1 RLHF Pipeline
1. **Data Curation**: оновлення датасетів, фільтри приватності, дедуплікація.
2. **Supervised Fine-Tuning (SFT)**: початкове навчання на схвалених відповідях.
3. **Reward Model Training**: тренування RM на людських або AI-оцінках (distillation).
4. **RL Optimization**: PPO/QLoRA з constrained reward.
5. **Evaluation Loop**: симулятори + людські перевірки + scorecards.
6. **Deployment Gate**: технічний та бізнес рев’ю.

### 10.2 RLAIF Інтеграція
- Використовувати узгоджені AI-оцінювачі як попередній шар, але з регулярним калібруванням на людських еталонах.
- Застосовувати active disagreement sampling: коли AI й людина сильно розходяться, кейси повертаються людині.
- Зберігати explainability звіти для кожного AI-оцінювача.

## 11. Виробничі перевірки (Pre-deployment)
- **Checklist**: функціональний, безпековий, комплаєнс, UX, доступність.
- **Red Teaming**: сценарії зловживань, конфлікт інтересів, маніпуляції.
- **Shadow Deployment**: A/B з поточною моделлю, відстеження метрик у реальному часі.
- **Rollback Plan**: автоматичні тригери на відкат при деградації KPI.

## 12. Регресійні тести
- **Data Regression**: контрольні набори промптів з фіксованими очікуваними відповідями/оцінками.
- **Reward Regression**: збереження метрик RM (AUC, calibration) та порівняння з попередніми релізами.
- **Policy Regression**: автоматичні QA-тести на політики (наприклад, forbidden content detection).
- **Модульні тести симуляторів**: верифікація, що сценарії ринку залишаються відтворюваними.

## 13. Контроль процесу та аудит
- Запровадити pipeline у MLOps-системі (наприклад, Prefect/Kubeflow) з логуванням та артефактами.
- Використовувати DVC/MLflow для версіонування моделей, даних, reward-функцій.
- Щоквартально проводити незалежний аудит (внутрішній/зовнішній) відповідності політикам та результатам.
- Документувати усі зміни у `docs/alignment/changelog.md`.

## 15. Подальший розвиток
- Розширення RLAIF на мультимодальні дані (аудіо, графіки, документи).
- Використання federated learning для приватних портфелів клієнтів.
- Впровадження human-in-the-loop в реальному часі під час критичних сесій трейдингу.
- Аналіз впливу моделей на бізнес KPI та ESG-показники.
